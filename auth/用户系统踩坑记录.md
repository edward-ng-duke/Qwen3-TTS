# 用户系统踩坑记录 — v1.4

> 记录 v1.4 集成用户认证系统（本地 JWT + ES 外部认证 + 影子用户同步）过程中遇到的坑点，供后续维护参考。

---

## 1. ES 查询字段类型不匹配

**现象**: ES Token 验证时查询无结果，明明 Token 存在于索引中却查不到。

**原因**: Elasticsearch 的 `tokens` 字段在不同环境中映射类型不一致——有的是 `keyword`，有的是 `text`。`term` 查询只能匹配 `keyword` 字段，`match` 查询才能匹配 `text` 字段。用错查询类型会静默返回空结果。

**解决方案**: 启动时自动探测字段映射类型（`_detect_query_strategy()`），选择对应查询方式。运行时如果主策略失败，自动回退到另一种策略：

```python
# es_auth_service.py
# 主策略失败时自动切换
if not user_data and self._fallback_strategy:
    user_data = await self._query_with_strategy(self._fallback_strategy, token)
```

## 2. MongoDB 索引名冲突

**现象**: 启动时报错 `OperationFailure: Index already exists with a different name`。

**原因**: 旧版本 MongoDB 自动创建了名为 `username_1` 的唯一索引，新版本代码想创建名为 `idx_username_unique` 的索引（相同字段、相同选项但名字不同），MongoDB 不允许同字段存在两个不同名的索引。

**解决方案**: `ensure_indexes()` 先检查是否已存在功能等价的索引（无论名字是什么），如果有就跳过创建：

```python
# user_repository.py
existing = await self.collection.index_information()
has_username_index = any(
    v.get("key") == [("username", 1)] for v in existing.values()
)
if not has_username_index:
    await self.collection.create_index("username", unique=True, name="idx_username_unique")
```

## 3. JWT 密钥必须两端一致

**现象**: 用户在 conference_ws（MyGPT）登录后访问 auto-writeh，提示未认证；反之亦然。

**原因**: 两个应用共享 `conference_ws.users` 集合，但如果 `JWT_SECRET_KEY` 不一致，一方签发的 Token 另一方无法解码验证。

**解决方案**: 部署时必须确保两个应用使用**完全相同**的 `JWT_SECRET_KEY`。已在 deploy_v14.md 和环境变量参考中标注。

## 4. 中间件异常导致 500

**现象**: 偶发的认证相关 500 错误，尤其在 ES 服务不稳定时。

**原因**: Auth Middleware 中 Token 解码、数据库查询、ES 请求等任何一步抛出未捕获异常，都会导致整个请求以 500 响应。

**解决方案**: 将整个认证流程包裹在 try/except 中，任何异常都降级为"未认证"（`request.state.user = None`），由后续的 `require_user` 依赖统一返回 401：

```python
# auth.py middleware
async def _authenticate(self, request):
    try:
        # ... 认证逻辑
    except Exception as e:
        logger.warning("auth_middleware_error", error=str(e))
        return None  # 降级为未认证，不 crash
```

## 5. 静态资源走认证造成性能浪费

**现象**: 每次加载页面，浏览器请求的 `.js`、`.css`、图片文件都触发一次 MongoDB 或 ES 查询。

**原因**: Middleware 对所有请求一视同仁，包括静态资源。

**解决方案**: 在 Middleware 入口处按路径后缀跳过认证：

```python
STATIC_EXTENSIONS = {".js", ".css", ".png", ".jpg", ".ico", ".svg", ".woff", ".woff2"}

if any(request.url.path.endswith(ext) for ext in STATIC_EXTENSIONS):
    return  # 跳过认证
```

## 6. JWT 不是无状态的 — 删除用户后旧 Token 仍可用

**现象**: 管理员删除某个用户后，该用户手持的未过期 JWT 仍然可以访问 API。

**原因**: 标准 JWT 是无状态的，只要签名和有效期正确就通过验证。删除用户不会使已签发的 Token 失效。

**解决方案**: 每次请求都额外查数据库验证用户是否仍然存在且为 active 状态。代价是每个请求多一次 MongoDB 查询，但对于当前用户规模（< 50）完全可以接受：

```python
# middleware 中 JWT 验证后追加
user = await user_repository.get_by_id(payload["sub"])
if not user or not user.is_active():
    return None  # Token 有效但用户已不可用
```

## 7. ES Token 提取的大小写和格式问题

**现象**: 前端通过 Cookie 或 Header 传递 ES Token，但后端偶尔提取不到。

**原因**:
- HTTP Header 名在不同代理/框架中大小写处理不一致（`X-Token-2` vs `x-token-2`）
- Cookie 值可能包含引号或空白
- 某些 Token 长度与预期不一致（不同 ES 版本生成的 Token 长度可能不同）

**解决方案**:
1. Header 查找时同时尝试原始名和小写名
2. Token 值做 strip 处理
3. 长度校验放宽为 ±10 字符容差而非精确匹配
4. 每一步都加详细日志，方便排查：

```python
logger.debug("es_token_extract", cookie_keys=list(cookies.keys()),
             header_keys=list(headers.keys()), token_prefix=token[:8] if token else None)
```

## 8. ES 全量同步分页排序字段不兼容

**现象**: 定时全量同步 ES 用户到本地 MongoDB 时，部分 ES 环境报排序字段不存在。

**原因**: 分页查询使用 `search_after` + `sort: ["username.keyword"]`，但某些 ES 环境中 `username` 字段没有 `.keyword` 子字段（取决于 mapping 配置）。

**解决方案**: 三级回退策略：
1. 先尝试 `username.keyword` 排序
2. 失败则回退到 `username` 排序
3. 再失败则放弃分页，直接拉取全量（size=10000 上限）

```python
for sort_field in ["username.keyword", "username"]:
    try:
        users = await self._fetch_sorted(sort_field)
        break
    except Exception:
        continue
else:
    users = await self._fetch_all_unsorted(size=10000)
```

## 9. 密码哈希迁移 — SHA256 与 bcrypt 共存

**现象**: 旧用户无法登录，或新创建的用户在旧系统上无法登录。

**原因**: 旧版本使用 SHA256 哈希（`hashlib.sha256`），新版本改用 bcrypt。两种哈希格式完全不同，直接替换会导致所有旧用户密码失效。

**解决方案**: 登录时自动检测并迁移：

```python
def verify_and_migrate(password, stored_hash):
    if stored_hash.startswith("$2b$"):
        return bcrypt.checkpw(password, stored_hash)  # 新格式
    else:
        if sha256(password) == stored_hash:            # 旧格式匹配
            new_hash = bcrypt.hashpw(password)         # 透明升级
            await update_password_hash(user_id, new_hash)
            return True
        return False
```

用户无感知，首次用旧密码登录后自动升级为 bcrypt。

## 10. 硬删除 vs 软删除 — 测试断言全部打脸

**现象**: 切换为硬删除后，所有相关测试用例失败。

**原因**: 旧版本删除用户是软删除（设 `status=disabled`），API 返回 200 + disabled 状态。新版本改为硬删除（从 MongoDB 中物理移除），删除后查询返回 404。测试断言还停留在旧行为。

**解决方案**: 统一更新测试期望：

```python
# 旧断言
assert response.status_code == 200
assert data["status"] == "disabled"

# 新断言
response = client.get(f"/api/users/{user_id}")
assert response.status_code == 404
```

## 11. Docker 构建上下文错误

**现象**: 容器启动后前端页面空白，控制台 404 错误加载不到 JS/CSS。

**原因**: 在 `docker/` 子目录下执行 `docker build .` 导致构建上下文不包含项目根目录的 `frontend/` 文件夹，COPY 指令静默跳过不存在的路径。

**解决方案**: 始终从项目根目录构建，用 `-f` 指定 Dockerfile 路径：

```bash
# 正确
docker build -f docker/Dockerfile -t auto-writeh:v1.4-amd64 .

# 错误
cd docker && docker build .
```

## 12. 异步任务未优雅关闭

**现象**: 容器停止时日志中出现 `Task exception was never retrieved` 警告。

**原因**: ES 定时同步任务使用 `asyncio.create_task()` 启动后台循环。应用关闭时 `asyncio.sleep()` 被取消，抛出 `CancelledError`，但没有被捕获处理。

**解决方案**: 在 lifespan 的 shutdown 阶段显式取消任务并捕获异常：

```python
# app.py lifespan
async def lifespan(app):
    sync_task = asyncio.create_task(_periodic_es_user_sync())
    yield
    sync_task.cancel()
    try:
        await sync_task
    except asyncio.CancelledError:
        pass  # 正常关闭，不告警
```

## 总结

| 类别 | 坑点 | 严重程度 |
|------|------|----------|
| ES 集成 | 字段类型不匹配、排序字段不兼容、Token 格式差异 | 高 |
| 数据库 | 索引名冲突、硬删除行为变更 | 中 |
| 认证 | JWT 密钥不一致、Token 非无状态、中间件异常 | 高 |
| 部署 | Docker 构建上下文、异步任务关闭 | 中 |
| 兼容性 | 密码哈希迁移、静态资源认证 | 中 |

> **经验教训**: 对接外部系统（ES）时不要假设字段类型和版本行为一致，多加回退策略和详细日志。认证中间件必须做到"出错不 crash"，宁可降级也不能 500。
